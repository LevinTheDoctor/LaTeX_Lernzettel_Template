\subsection{Serverless}
\textbf{Serverless} bezeichnet in der Cloud die Möglichkeit, Code und Services
auszuführen, ohne selbst einen Server bereitzustellen oder zu verwalten. Man
kümmert sich also weder um das Betriebssystem noch um Skalierung oder die
darunterliegende Hardware. Stattdessen stellt man lediglich den Code bereit
(\enquote{deployt} ihn), und der Cloud-Anbieter führt ihn aus. Abgerechnet
wird \emph{nur die tatsächliche Nutzung}.

% LEKTORAT (fachlich): Der ursprüngliche Satz bezog das Deployment direkt auf
% ein Docker-Image samt Registry. Für Serverless/FaaS ist ein Container-Image
% aber nicht zwingend – häufig wird nur der Code deployt. Der Bezug wurde daher
% entkoppelt; das Thema CI/CD bleibt erhalten, das Detail zu Images/Registry
% gehört sachlich ins Docker-/Kubernetes-Kapitel.
Das Deployment lässt sich über eine CI/CD-Pipeline automatisieren – etwa so,
dass ein Git-Commit auf einen bestimmten Branch automatisch die neue Version
bereitstellt. Manche Plattformen deployen dabei den Code direkt, andere
paketieren ihn zuvor in ein Container-Image.

\subsection{Function as a Service – FaaS}
% LEKTORAT (fachlich, Unsicherheit gekennzeichnet): "Teil des PaaS-Modells"
% abgeschwächt. Die Einordnung von FaaS ist in der Literatur uneinheitlich:
% Viele rechnen es PaaS zu, andere sehen FaaS (zusammen mit Serverless) als
% eigenständiges Cloud-Computing-Modell. Falls die Vorlesung FaaS ausdrücklich
% als PaaS einordnet, bitte an die Kurssprache anpassen.
\textbf{Function as a Service (FaaS)} wird meist dem PaaS-Modell zugerechnet und ist
besonders eng mit Serverless verwoben – FaaS ist gewissermaßen die bekannteste
Ausprägung von Serverless. Bei FaaS wird Code mit Business-Logik – also
Backend-Code – in der Cloud bereitgestellt, meist in Form einzelner Funktionen.

Diese Funktionen werden automatisch horizontal skaliert und durch \textit{Events}
ausgelöst, beispielsweise durch HTTP-Requests, Webhooks oder Ereignisse
anderer Cloud-Dienste (z.\,B. ein neuer Datei-Upload in einen AWS-S3-Bucket).
% LEKTORAT (fachlich): "AWS S3" als Trigger präzisiert – nicht der Dienst
% selbst, sondern ein S3-Event (z. B. ein Objekt-Upload) löst die Funktion aus.
Das Deployment ist denkbar einfach, da im Kern nur der Funktions-Code abgelegt
werden muss.

FaaS-Dienste heißen bei Azure \emph{Azure Functions}, bei AWS \emph{AWS Lambda}
und bei Google \emph{Google Cloud Functions}. Open-Source-Alternativen sind
OpenFaaS, Nuclio, Fn, Fission und Apache OpenWhisk.

\subsubsection{Limitationen von Serverless}
So komfortabel Serverless ist – es gibt klare Grenzen, die man bei der
Architektur-Entscheidung kennen sollte:
\begin{itemize}
  \item \textbf{Cold Starts:} Wird eine Funktion längere Zeit nicht
        aufgerufen, muss die Laufzeitumgebung erst neu hochgefahren werden.
        Der erste Aufruf ist dadurch spürbar langsamer.
  \item \textbf{Begrenzte Ausführungszeit:} Funktionen dürfen nur eine
        maximale Laufzeit haben (bei AWS Lambda z.\,B. 15 Minuten). Für lang
        laufende Verarbeitungen ist FaaS daher ungeeignet.
  \item \textbf{Zustandslosigkeit:} Funktionen sind von Natur aus stateless.
        Zustand muss extern gehalten werden, etwa in einer Datenbank oder
        einem Object Store.
  \item \textbf{Eingeschränktes lokales Testen:} Trigger und abhängige
        Cloud-Dienste lassen sich lokal nur schwer nachbilden (siehe auch die
        Phase \enquote{Lokaler Test} weiter unten).
  \item \textbf{Vendor Lock-in:} Funktionen sind oft eng an die Dienste eines
        bestimmten Anbieters gekoppelt, was einen Wechsel erschwert.
\end{itemize}

\subsubsection{FaaS und Abrechnung: Das Double-Spending-Problem}
FaaS wird nach dem Pay-as-you-go-Prinzip abgerechnet, häufig sogar pro
Millisekunde Laufzeit (\enquote{Pay-by-the-Millisecond}). Die Ausführungszeit
einer Funktion schlägt sich also direkt in den Kosten nieder – ihre
Optimierung lohnt sich unmittelbar.

Genau hier lauert eine typische Kostenfalle:
% LEKTORAT (fachlich/Begriff): Beschriebenes Phänomen ist korrekt, der Begriff
% jedoch heikel. In der Praxis/Literatur ist für dieses Problem eher der Begriff
% "Double Billing" gebräuchlich; "Double Spending" stammt ursprünglich aus dem
% Blockchain-Kontext (dieselbe Einheit zweimal ausgeben). Falls der Begriff aus
% der Vorlesung stammt, hier belassen – der Hinweis dient nur der Einordnung.
% Außerdem präzisiert: Das Problem entsteht beim SYNCHRONEN Aufruf (Warten),
% nicht generell, wenn eine Funktion eine andere aufruft.
\begin{warnbox}[Double-Spending-Problem]
  Ruft eine Funktion \emph{synchron} eine zweite Funktion auf und wartet auf
  deren Ergebnis, laufen für die Dauer des Aufrufs \textbf{beide} Funktionen
  gleichzeitig: Die aufrufende Funktion wartet (und kostet), während die
  aufgerufene arbeitet (und ebenfalls kostet). Man bezahlt also doppelt.
\end{warnbox}
Vermeiden lässt sich das beispielsweise durch \emph{asynchrone} Aufrufe
(\enquote{fire and forget}) oder durch Orchestrierungsdienste wie AWS Step
Functions, die das Warten übernehmen, ohne dass die aufrufende Funktion dafür
Rechenzeit verbraucht.

\subsubsection{Entwicklung von FaaS}
Die Entwicklung einer Funktion durchläuft typischerweise sechs Phasen:
\begin{process}
  \step{\textbf{Design:} Entwurfsphase, in der festgelegt wird, was die
        Funktion tun soll. Schon hier sollte man typische FaaS-Fallstricke
        berücksichtigen – etwa das weiter oben beschriebene
        Double-Spending-Problem.}
  \step{\textbf{Code:} Das Design wird in Code umgesetzt.}
  \step{\textbf{Lokaler Test:} Der Code wird in der lokalen Umgebung getestet,
        meist nur daraufhin, ob er das erwartete Ergebnis liefert, fehlerfrei
        durchläuft und welche Laufzeit er hat. Weitergehende Tests sind bei
        Serverless oft schwierig (siehe Limitationen).}
  \step{\textbf{Integrationstests:} Tests in einer echten Cloud-Umgebung,
        idealerweise in einer eigenen Test- bzw. Staging-Umgebung.}
        % LEKTORAT (fachlich): Vorher "meist auch in einer Produktionsumgebung".
        % Integrationstests gehören üblicherweise in eine separate Test-/Staging-
        % Umgebung, nicht in Produktion. ("Testing in Production" wird bei
        % Serverless zwar diskutiert, ist aber ein Sonderfall, keine Regel.)
  \step{\textbf{Deploy:} Den Code in die Cloud bringen.}
  \step{\textbf{Monitoring:} Beobachten, wie die Funktion in der Cloud arbeitet
        und performt (Output, Laufzeit, Fehler). Daraus ergeben sich häufig
        weitere Anpassungen – der Zyklus beginnt von vorn.}
\end{process}

FaaS abstrahiert den Server vollständig weg. Wer hingegen mehr Kontrolle über
die Laufzeitumgebung braucht – etwa eigene Abhängigkeiten oder lang laufende
Dienste –, greift häufig zu Containern. Wie diese funktionieren, zeigt das
nächste Kapitel.
